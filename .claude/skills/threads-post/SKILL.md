# 스레드 글쓰기 스킬 — junominu / @solopreneur.jm

하성재님의 스레드 글을 **냉정한 스레드 분석가**의 관점에서 설계한다.
목표는 "예쁜 글"이 아니라 **초반 60~90분 안에 알고리즘을 통과하고 전환(답글·프로필 클릭·팔로우)을 만드는 글**이다.

---

## 0. 이 스킬의 입·출력

- **입력(사용자가 주거나 물어서 확정)**: ① 대상 프로젝트(slug) 또는 주제, ② 글 유형(앵커 제작기 / 단발 / 훅만), ③ 강조하고 싶은 사연·각도(선택).
- **출력**: §10 형식대로 — 복붙 가능한 스레드 초안 + 훅 대안 3개 + 냉정 채점 + 발행 팁.
- **원재료는 반드시 `src/projects.js`에서 읽는다.** 숫자·기능·막힌 점을 지어내지 말 것. 없으면 사용자에게 묻는다.
- **발행 로그 `threads-log.md`(이 스킬 폴더)를 먼저 읽고, 초안을 낸 뒤 갱신한다.** 이미 쓴 각도·훅·톤을 참고해 중복·방향 이탈을 막는다(§11).

### 폴더 4단 구조 (역할이 다르다)

| 파일 | 역할 | 언제 |
|---|---|---|
| `SKILL.md`(이 문서) | **글 한 편**을 어떻게 쓰나 — 훅·구조·채점 | 초안 요청 때 |
| `threads-strategy.md` | **채널을 어떻게 굴리나** — 플랫폼 팩트·리듬·답글·프로필·KPI | 운영/전략 질문, 또는 축·토픽 판단이 필요할 때(§12) |
| `threads-log.md` | **뭘 썼나** — 이력·반응·다음 각도 | 글 쓰기 전·후 |

⚠️ **플랫폼 사실이 두 문서에서 어긋나면 `threads-strategy.md` §1이 우선**이다(그쪽이 기준일과 출처를 갖는다).

## 1. 냉정한 전제 — 스레드는 왜 "극과 극"인가

스레드 알고리즘 ≈ **씨앗 뿌리고 증폭(seed-and-amplify)**. 글을 먼저 **팔로워의 2~5%** 에게만 노출하고, **초반 60~90분 신호**(완독·좋아요·**답글**·프로필 클릭·리포스트)가 좋으면 폭발적으로 밀어주고, 약하면 그대로 죽인다. 그래서 결과가 대박 아니면 폭망으로 갈린다. 이 스킬의 모든 규칙은 이 한 문장에서 나온다:

1. **첫 줄이 결과의 80%.** 폭망하는 글의 90%는 첫 줄이 근황·맥락설명·빌드업이다. 첫 줄은 3초 안에 긴장/궁금/공감을 만들어야 한다.
2. **답글 > 좋아요.** 스레드는 "대화"를 가장 세게 가중한다. Mosseri 본인이 *"한 계정의 답글 총합은 게시물 총합만큼 가치 있다"* 고 말했다. CTA는 좋아요가 아니라 **댓글을 유발**해야 한다(질문·공감 요청·가벼운 논쟁).
3. **완독률이 연결 스레드를 살린다.** 각 편은 **다음 편을 당기는 미끼(open loop)** 로 끝나야 한다. 1편에서 이탈하면 나머지는 존재하지 않은 것과 같다.
4. **감정 > 정보.** 정보만 있으면 저장은 돼도 확산은 안 된다. 정보를 **사연에 싸서** 준다.
5. **한 글 = 한 메시지.** 두 개를 섞으면 둘 다 죽는다.
6. **한 글로 판단하지 않는다.** 초반 30개는 캘리브레이션이다. 양 + 반복으로 감을 잡는다.
7. **골든타임을 쓴다.** 올린 뒤 **60~90분**간 모든 답글에 즉답 → 알고리즘 연료.
8. **"이 사람 누구지?"를 만들면 한 번 더 밀어준다.** **프로필 클릭이 공식 랭킹 신호**다(좋아요 확률·답글 참여·팔로우 확률·프로필 클릭·스크롤패스 5종 중 하나). 글이 사람에 대한 궁금증을 남기면 도달과 전환이 같이 오른다.
9. **팔로워 수는 도달을 거의 못 만든다.** 초기 답글 속도가 지배한다 → 팔로워는 목표가 아니라 결과다(`threads-strategy.md` §7).
10. **노골적 참여 유도는 역효과.** "좋아요 눌러주세요"류 engagement bait는 가시성이 축소된다. 질문은 **진짜 궁금한 것**이어야 한다.

> 기준일 **2026-07-22**. 플랫폼 사실의 원본·출처는 `threads-strategy.md` §1. 분기 1회 재확인하고, 바뀌면 두 문서를 같이 고친다.

## 2. 하성재님 포지셔닝 (절대 흔들지 않는 축)

- **정체성**: IT 회사에서 **22년**(서강대 컴공 → LG CNS **시스템 아키텍트** → 교보라이프플래닛생명 **정보보안 기획** → 공인중개사)을 일했지만 **제 손으로 제품을 끝까지 만들어 배포한 적은 없던 사람**이, 2026년부터 **AI 바이브코딩으로 혼자 기획·개발·배포**하는 1인 메이커.
  - ⚠️ "코딩 1도 몰랐다" 식 과장 금지(컴공 출신이라 거짓). 대신 **"IT 회사에 22년 있었지만, 제 손으로 제품을 끝까지 만들어 본 적은 없었다"** — 이게 진실이고 더 강하다.
  - ⚠️ **과거 업무를 "설계"라는 말로 요약하지 말 것**(2026-07-22 본인 확인). 업무는 **시스템 아키텍트 · 정보보안 기획**으로 그대로 쓴다. "설계는 오래 했지만…" 류의 표현 **금지**.
  - 확실한 사실: 서강대 컴퓨터공학과 · **IT 22년**(LG CNS 1999.12–2010.06 시스템 아키텍트 / 교보라이프플래닛생명 2011.01–2022.10 정보보안 기획) · 공인중개사(2023.06–2025.12) · 2026년부터 1인 메이커. **그 밖의 업무 디테일은 본인에게 확인 후 쓴다.**
- **차별점**: 만드는 게 전부 **"진짜 누군가의 불편"** 이다 — 발달장애 아이 부모, 1인 공인중개사, 치매 걱정 어르신, 녹취에 지친 실무자. → **따뜻함 + 실전**이 무기.
- **관문(목표)**: 인지도 → 앱 판매 · 강의 · **1:1 코칭** · **외주**.
- **톤**: 따뜻한 존댓말 구어체(`~어요/~습니다` 혼용). 이모지 2~3개.
- **금지**: 허슬브로(돈 자랑·과장·낚시), 약자·부모 대상 자극/공포 마케팅, 의료적 단정("치료된다"류), 없는 성과 지어내기.

## 3. 시그니처 구조 — "제작기 앵커 아크" (프로젝트당 1개, 고정글감)

프로젝트마다 **딱 하나** 만드는 대표 연결 스레드. PAS(문제-증폭-해결) + Behind-the-Scenes(어떻게 생각했나)를 합친 6비트.

| 비트 | 역할 | 규칙 |
|---|---|---|
| ① 훅 | 한 사람의 **구체적 장면** 한 줄 | §5 훅 라이브러리에서. 통계·일반론 금지, 한 사람·한 장면 |
| ② 증폭 | 그 문제가 얼마나 흩어지고 힘든지 | PAS의 Agitate. 독자가 "내 얘기다" 하게 |
| ③ 전환점 | "곁에서 보다 못 참고, 그래서 만들기로" | 왜 하필 내가. 사명감/개인적 계기 |
| ④ 만든 것 | 무엇을·어떻게 | 기능 **3개 이내**, 자랑 아니라 해결로 |
| ⑤ 진짜 이야기 | **막혔던 점 1~2개** | ★신뢰의 핵심. 남들이 안 보여주는 실패/우회. `detail.challenges`에서 |
| ⑥ 의미 + CTA | 누구에게 도움 되면 좋겠다 + **질문** + 링크 | CTA는 댓글 유발형. 링크는 **완독률 보호를 위해** 마지막 편에(페널티 회피가 아님, §9) |

- 각 편은 150자 안팎, 첫 편은 특히 짧고 세게.
- ①~③은 **문제·사연**으로 채우고 ④~⑥에서 **제작 과정**으로 착지 = 하성님이 원한 전개.
- ②→③, ④→⑤ 사이에 open loop("근데 여기서 문제가 생겼어요" 같은)로 다음 편을 당긴다.

### 3-B. 대안 아크 — "실험 중계" (결과를 먼저 깔고 과정을 중계)

제작기 앵커가 **문제→해결**이라면, 실험 중계는 **"내가 X를 시켜봤다 → 실황 → 결과 → 회고"**다. 빌드를 **실험으로 프레이밍**하면 완독률이 크게 오른다(외부 검증: `winning-patterns.md` 분해 #1 — 이 아크의 대표 사례가 단발 데모보다 도달이 수십 배였다).

| 비트 | 역할 | 규칙 |
|---|---|---|
| ① 훅 | **결론-먼저**(§5) | "AI한테 ~ 시켰더니 ~ 됐다"를 **1편에서 스포일**. 독자는 결과가 아니라 "어떻게"가 궁금해 끝까지 본다 |
| ②~④ 실황 | 시간·단계 순 중계 | **실패·갈아엎음·반전을 그대로.** ★신뢰의 핵심. beat 단위로 |
| ⑤ 회고 | 메타 관찰 한 줄 | "해보니 결이 다르더라" 같은. 숫자보다 **감정·의미**로 |
| ⑥ 착지 | 배운 것 + 질문 CTA + (선택)결과물 | 링크는 맥락에 싸서 마지막에 |

- 소재: `detail.timeline`(단계)·`detail.challenges`(실패)·`detail.prompts`(실제 지시). **"AI에게 시켰다"는 실제로 시킨 것만** — 과장 금지.
- ⚠️ **매출·완판 프레임 금지**(§2). 실험의 목표는 **기능/완성·되더라·안 되더라**이지 돈 액수가 아니다.
- 앵커 제작기와 마찬가지로 무거운 포맷 → **월 1~2회**. 나머지는 §4 로테이션.

## 4. 콘텐츠 로테이션 — 매일 앵커만 쓰면 안 된다

앵커 아크는 무겁다. 매일 쓰면 본인도 독자도 지치고 알고리즘도 싫어한다. **"문제·공감 훅"은 시그니처 오프닝으로 유지하되 다양한 유형에 얹는다.** 권장 비율(채널 축 배분은 `threads-strategy.md` §4와 동일):

| 유형 | 비중 | 미니 템플릿 |
|---|---|---|
| 🔧 **빌드 로그**(제작 뒷이야기·막힌 점·실제 프롬프트) | 35% | "이거 만들다 ○○에서 막혔어요" → 어떻게 우회 → 배운 점 (`detail.challenges`·`detail.prompts`). building 3개가 무한 소재 |
| 🫶 문제·공감 사연 (**시그니처**) | 25% | 한 장면 훅 → 공감 → 짧은 생각 → 질문 |
| 💡 바이브코딩 인사이트·how-to | 25% | "비개발자가 ○○ 붙인 법" → 3단계 → 저장 유도 (코칭·강의로 직결). **장문 첨부 최적**(§9) |
| 📈 마일스톤·지표 | 15% | "7개월, 혼자, 서비스 8개" 같은 Data Story |

❓ 질문·참여 유도는 **독립 유형이 아니라 모든 글의 끝 CTA**로 상시 넣는다(§8 채점표).

**토픽 기본값은 `#바이브코딩`.** 스레드 한국 코어는 20대(전체의 40%)라 **메이커 각도의 도달이 부모 각도보다 좋다.** `#발달장애` 등 제품 타겟 토픽은 🫶 사연 축에서만 선택적으로 쓴다. 근거와 예외는 `threads-strategy.md` §1·§4.

### 형식 — 기본은 1편. 후킹·공감이 편 수보다 먼저다

**북극성 둘: ① 얼마나 후킹되나(첫 줄) ② 얼마나 공감되나.** 편 수·형식·아크는 전부 이 둘을 위해 존재한다. **1편이냐 10편이냐는 핵심이 아니다** — 덜 후킹되고 덜 공감되는 4편보다, **강하게 후킹되고 깊게 공감되는 1편이 이긴다.**

- **기본값 = 1편.** 강한 훅 + 공감 한 스푼 + **이미지 1장** + **CTA(진짜 질문 + 링크)**. 데모·근황·인사이트·사연·마일스톤 — 거의 다 1편으로 끝낸다.
- **여러 편(연결 스레드)은 예외.** 이야기에 **정말로** 순차 비트(실황→실패→반전→회고)가 있어 각 편이 open loop로 다음을 당길 때만. 그게 실험 중계(§3-B)·제작기 앵커(§3)이고, **월 1~2회짜리 이벤트**다.
- **억지 4등분 금지.** 1편에 다 담기면 1편으로. 미끼가 없는데 늘리지 말고, 진짜 비트가 있는데 뭉개지도 말 것.

> 검증(`winning-patterns.md` 분해 #1): 윤자동도 **평소엔 1편**(데모·근황), 대표 바이럴만 ~12편. **중간값(고정 4편)은 안 쓴다.**

## 5. 훅 라이브러리 (가장 중요) — 첫 줄 공식

첫 줄은 항상 **3개 이상 만들어 가장 센 것**을 고른다. 공식별 스켈레톤 + 하성님 예시:

- **장면형**(제일 강함): 시간·인물·행동을 한 컷으로.
  예) "새벽 2시, 엄마는 아직도 맘카페에서 치료실을 검색하고 있었어요."
- **반전형**: 통념/과거 ↔ 현재의 낙차.
  예) "IT 회사를 22년 다녔지만, 제 손으로 만든 서비스는 하나도 없었습니다."
- **숫자충격형**: 맥락 없는 숫자로 긴장.
  예) "혼자서, 6개월, 서비스 8개. 개발자 0명, 투자 0원."
- **상식뒤집기형**: "다들 X라던데 사실 Y."
  예) "'비개발자는 앱 못 만든다'는 말, 반은 맞고 반은 틀렸어요."
- **고백/실패형**: 약점을 먼저.
  예) "출시하려는데, 코드보다 서류가 더 큰 벽이었습니다."
- **질문형**(답글 유발): 독자의 경험을 직격.
  예) "바우처 카드 받고 '이걸 대체 어디서 쓰지?' 막막했던 적 있으세요?"
- **손실회피형**: 놓치고 있는 것.
  예) "정보를 몰라서 못 받고 지나가는 지원사업이, 생각보다 많습니다."
- **결론-먼저형**(연결 스레드에 강함): 결과를 1편에서 **먼저 까발리고**, 과정을 미끼로 완독을 끈다. 독자는 결과가 아니라 **"어떻게"** 가 궁금해 끝까지 본다. §3-B 실험 중계 아크의 기본 훅.
  예) "결론부터. 이 기능, AI한테 시켰더니 반나절 만에 붙었어요. 근데 그 반나절에 세 번 갈아엎었습니다."
  ⚠️ 스포일하는 결과는 **숫자·상태**(커밋·일수·되더라/안 되더라)여야지 **매출 자랑이 아니다**(§2). 근거·이식 규칙: `winning-patterns.md` 분해 #1.

훅 금지: 인사("안녕하세요"), 근황("오늘 ~다녀왔어요"), 맥락설명으로 시작, 물음표 없는 밋밋한 선언.

## 7. 생성 워크플로 (이 순서를 지킨다)

1. **`threads-log.md`를 읽는다(§11).** 이 프로젝트에 이미 앵커가 있는지, 어떤 훅·각도를 썼는지, "아직 안 쓴 각도" 후보가 있는지 확인 → 중복·방향 이탈 방지.
2. `src/projects.js`를 읽어 대상 프로젝트 `detail` 확보(없으면 사용자에게 사연·숫자 확인).
3. 글 유형 결정: **앵커 제작기(§3)** / **실험 중계(§3-B)** / **단발(§4)** / **훅만(§5)**. (앵커가 이미 있으면 단발/인사이트/실험 중계 쪽으로 유도)

   - 축·토픽·형식 판단이 애매하면 **`threads-strategy.md` §4·§6을 읽고 정한다**(§12).
   - 훅·아크가 밋밋하거나 "○○처럼 쓰고 싶다"는 요청이면 **`winning-patterns.md`의 검증된 기법**을 재료로 쓴다(그대로 복제 말고 §2 톤으로 이식).
4. **훅 3개 생성(§5)** → 가장 센 것 선택(나머지는 대안으로 제시). 로그에 이미 쓴 훅과 겹치지 않게.
5. 본문 조립: 앵커는 §3 6비트, 단발은 §4 미니 템플릿.
6. **냉정 채점(§8).** 60점 미만이면 **훅부터 다시**. 통과할 때까지 스스로 깎는다.
7. §10 형식으로 출력.
8. **`threads-log.md`에 이번 글 항목을 추가한다(§11).** 상태는 `초안`, "아직 안 쓴 각도"도 함께 기록.

## 8. 발행 전 냉정 채점표 (통과 못 하면 내보내지 않는다)

항목당 10점 · 13항목 **130점 만점 → 100점으로 환산**. **애매하면 폭망 쪽으로 판정한다.** 70점 미만이면 고쳐서 다시.

> ★ **최우선 2항목 = 후킹(첫 줄)·공감(감정에 실림).** 나머지가 다 좋아도 이 둘이 약하면 폭망이다 — 점수와 별개로 이 둘이 약하면 **훅부터 다시**. **1편 글이면 '(연결 스레드)' 항목은 빼고 환산**(기본은 1편, §4).

- [ ] 첫 줄이 3초 안에 긴장/궁금/공감을 만드나? (아니면 곧장 폭망)
- [ ] 한 글에 메시지가 하나인가?
- [ ] (연결 스레드) 각 편이 다음 편을 당기나(open loop)?
- [ ] (연결 스레드) 1편이 **결론·결과를 먼저 던지거나** 강한 미끼를 걸어 다음 편을 안 열 수 없게 하나? (결론-먼저 훅 §5 / 실험 중계 §3-B)
- [ ] CTA가 **진짜 궁금한 질문**으로 댓글을 유발하나? (「좋아요 눌러주세요」·「저장하세요」류 engagement bait는 **0점** — 가시성이 오히려 축소된다)
- [ ] 답글이 **오고 싶게** 열려 있나? (되물음·솔직한 빈틈·가벼운 위트로 대화 여지 — "좋은 글" 소리가 아니라 실제 답글이 나올 틈)
- [ ] **"이 사람 누구지?"를 남기나?** (프로필 클릭 = 공식 랭킹 신호이자 퍼널 입구)
- [ ] 정보가 감정/사연에 실려 있나?
- [ ] 과장·낚시·허슬브로 톤이 없나?
- [ ] 하성님 톤(따뜻한 존댓말 구어체)과 일치하나?
- [ ] 첫 편이 150자 안팎으로 짧나?
- [ ] 숫자·고유명사가 `projects.js` 실데이터인가?
- [ ] 토픽(해시태그)은 1개인가? (링크가 있다면 **맥락에 싸여** 있나?)

채점은 **정직하게**. "이 정도면 되겠지"는 폭망의 신호다.

## 9. 발행 운영 팁 (초안과 함께 안내)

- 첫 편을 올리고 **본인 답글로 나머지 편을 이어** 붙인다(= 연결 스레드). **답글도 예약 발행이 되므로** 뒷편은 미리 걸어두고 사람 상대는 실시간으로 해도 된다.
- **편 번호(1/7 등)는 스레드가 자동으로 붙인다 → 본문에 `[1/7]`을 직접 타이핑하지 말 것**(겹쳐서 두 번 나온다). 번호보다 **각 편 끝의 open loop(다음 편 미끼)** 가 완독률에 훨씬 중요하다.
- **토픽(해시태그)은 1개만** (기본 `바이브코딩`, §4). 남발은 도움이 안 된다.
- **대표 앵커는 프로필에 고정**(핀). 새 방문자 전환에 계속 일한다. 더 좋은 앵커가 나오면 교체.
- **발행량: 하루 1개 상한, 주 3~5개.** ⚠️ 하루에 여러 개 올리면 **새 글이 이전 글의 답글 누적을 끊는다.** 주 2개 미만이면 모멘텀이 안 붙는다. 시간대는 평일 밤.
- 올린 뒤 **60~90분 답글 즉답**(골든타임). 이게 알고리즘 연료다.
- **링크는 금지가 아니라 "맥락에 싸서".** 2024년의 링크 억제는 **리밸런싱돼 사라졌다**(Mosseri: URL 포함 글에 오히려 더 높은 값). 다만 **맥락 없는 맨 링크는 여전히 저조.** 앵커는 완독률 보호를 위해 링크를 마지막 편에 둔다.
- **장문이 필요하면 "텍스트 첨부"(최대 10,000자)** — 본문 500자 한도는 그대로다. 💡 인사이트·프롬프트 전문 공개에 최적이고, **앵커 제작기에는 쓰지 말 것**(편별 답글·완독 신호가 안 쌓인다). 판단표는 `threads-strategy.md` §6.
- **스크린샷 1장**(`public/shots/…`) 첨부 시 체류·조회 상승. 어울리는 캡처를 추천할 것.
- 글은 삭제 대신 **보관**. 소통(좋아요·답글) 활발히 = 계정 점수↑.
- **글을 안 쓰는 날에도 답글은 단다.** 답글이 채널 성장의 최대 지렛대다 → 전략은 `threads-strategy.md` §5.

## 10. 출력 형식 (항상 이렇게 답한다)

```
📌 유형: (앵커 제작기 / 단발-○○ / 훅)  ·  대상: (프로젝트명)

── 스레드 초안 ── (각 편을 개별 코드블록으로. 번호는 코드블록 "밖" 제목으로만 표기)
▼ 1편
[붙여넣을 본문만 — 편 번호 절대 넣지 말 것, 스레드가 자동 부여]
▼ 2편
[붙여넣을 본문만] … (편 사이 open loop로 다음을 당긴다)

🎣 훅 대안 3개
1) …
2) …
3) …

🧊 냉정 채점: 82/100
   - 가장 약한 고리: (한 줄)
   - 더 세게 하려면: (한 줄)

🚀 발행 팁: 토픽 #… / 시간대 / 고정 여부 / 추천 첨부 이미지(있으면 /shots 경로)
```

출력은 항상 **복붙 우선**(글 본문은 코드블록), 설명은 짧게. 하성님은 이 초안을 바로 스레드에 붙여 쓴다.
각 코드블록엔 **붙여넣을 텍스트만** 담고 편 번호는 넣지 않는다(스레드가 자동으로 붙임).

## 11. 발행 로그(`threads-log.md`) — 연속성 유지

같은 폴더의 `threads-log.md`는 **이미 쓴 글의 각도·훅을 기억**해, 다음 글이 반복되거나 톤에서 새는 걸 막는 장치다.

- **글 쓰기 전(워크플로 1단계)**: 로그를 읽어 ① 이 프로젝트에 앵커가 이미 있는지(있으면 새 앵커 대신 단발/인사이트), ② 어떤 훅·실데이터를 이미 썼는지, ③ "아직 안 쓴 각도" 후보가 있는지 확인한다.
- **초안을 낸 뒤(워크플로 8단계)**: 로그 맨 아래 `<!-- 새 항목은 이 줄 위에 추가 -->` 앞에 항목 1개를 추가하고, 상단 "프로젝트별 앵커 현황" 표도 갱신한다.
- 사용자가 "발행했다"고 하면 해당 항목 `상태: 초안` → `발행`으로 바꾸고 반응 메모를 채운다.

**항목 형식(그대로 따를 것):**
```
### YYYY-MM-DD · <프로젝트명> · <유형> · 축: 🔧|🫶|💡|📈 · 상태: 초안
- **한 줄 메시지**: (이 글이 남기려는 단 하나의 메시지)
- **선택한 훅(공식형)**: "..." ← §5 공식명 병기 (장면형/반전형/…)
- **아크/구성**: (편 순서 요약)
- **쓴 실데이터**: (projects.js에서 인용한 숫자·고유명사)
- **토픽**: #...
- **첨부 추천**: /shots/... · 장문첨부 사용: 예/아니오
- **아직 안 쓴 각도(다음 후보)**: (challenges·prompts·timeline에서 남은 글감)
- **반응 메모(발행 후 채우기)**: 조회 — · 답글 — · 팔로우 — · 사이트 유입 —
```

**축**과 **훅 공식명**을 반드시 적는다. 이 두 개가 있어야 나중에 "어떤 축·어떤 공식이 이기는지" 판정(아웃라이어 재활용, `threads-strategy.md` §8)이 가능하다.

## 12. 채널 운영은 `threads-strategy.md`로

이 스킬은 **글 한 편**을 담당한다. 아래 질문이 나오면 **`threads-strategy.md`를 읽고 답한다** — 이 문서에 답이 없다:

| 질문 | 문서 |
|---|---|
| "요즘 스레드 어떻게 돌아가?" 플랫폼 사실·수치 | §1 (기준일·출처 포함) |
| "X 하는 것처럼 하고 싶다" 이식 판단 | §2 |
| 프로필 소개글·고정글을 뭘로? | §3 |
| 뭘 얼마나 자주 올려야 하나 / 어떤 축 / 어떤 토픽 | §4 |
| **답글 어디에 어떻게 다나** (최대 지렛대) | §5 |
| 장문 첨부(10,000자)를 쓸까 말까 | §6 |
| 뭘 지표로 보나 / 계측이 되나 | §7 |
| 터진 글을 어떻게 우려먹나 | §8 |
| 90일 계획 | §9 |

**플랫폼 사실이 두 문서에서 어긋나면 `threads-strategy.md` §1이 우선**이고, 고칠 땐 두 문서를 같이 고친다.
