> 이 글은 초안입니다. 내 실제 경험과 표현으로 다듬을 예정.

혼자 여러 개의 사이트를 만들고 운영하다 보면 깨닫는 게 하나 있다. **나를 가장 지치게 하는 건 "새 기능"이 아니라 "작은 수정"이다.** 프로젝트 하나를 추가하려고 매번 컴포넌트를 새로 짜고, 레이아웃을 손보고, 링크를 연결해야 한다면, 두세 개만 돼도 손이 안 간다.

그래서 나는 프로젝트를 시작할 때 거의 항상 같은 질문부터 한다. **"이걸 나중에 고칠 때, 코드가 아니라 데이터만 고치게 할 수 있나?"**

## 카드 하나 = 객체 하나

이 작업실(junominu.com)이 가장 단순한 예다. 새 프로젝트가 생기면 배열에 객체 하나를 추가한다. 그게 전부다.

```js
{
  name: "프로젝트 이름",
  emoji: "✨",
  description: "한 줄 설명.",
  url: "https://...",
  status: "live",        // live | building | idea
  tags: ["태그1", "태그2"],
}
```

화면은 이 배열을 그릴 뿐이다. 카드의 모양, hover 효과, 상태 색깔은 한 번만 만들어 두면 끝. 다음부터 나는 "디자인"이 아니라 "내용"만 생각하면 된다. AI에게도 "이 배열에 객체를 하나 더 추가해줘"라고 말하는 게, "새 카드 컴포넌트를 만들어줘"보다 훨씬 사고가 적다.

## 상세 페이지도, 블로그도 같은 원리

이 원칙은 위로 갈수록 힘을 발휘한다.

- **프로젝트 상세 페이지**는 각 프로젝트의 `detail` 객체(요약·기능·스택·타임라인·회고)를 읽어 **있는 필드만** 자동으로 그린다. 채우고 싶은 것만 채우면 된다.
- **블로그**는 폴더에 마크다운 파일 하나를 넣으면 글이 된다. 목록도, 글 페이지도, 정렬도 알아서 된다.
- **시간순 기록(저니)**도 따로 관리하지 않는다. 각 프로젝트의 시작일을 읽어 자동으로 줄을 세운다.

공통점은 하나다. **데이터가 진실의 원천(single source of truth)이고, 화면은 그걸 비추는 거울일 뿐이다.** 내가 한 곳을 고치면 관련된 화면이 전부 따라온다. 같은 정보를 두 군데에 적는 일이 사라진다.

## 비개발자에게 특히 유리한 이유

문법을 깊이 모르는 사람일수록 이 구조가 유리하다고 느낀다.

1. **고칠 곳이 한 곳이다.** "어디를 바꿔야 하지?"로 헤매지 않는다. 데이터 파일만 열면 된다.
2. **AI에게 시키기 쉽다.** "이 객체에 필드 하나 추가" 같은 작은 지시는 실패가 적고, 결과를 눈으로 확인하기도 쉽다.
3. **무너뜨리기 어렵다.** 화면 로직은 안 건드리니, 데이터를 잘못 적어도 그 카드 하나만 어색해질 뿐 사이트 전체가 깨지지 않는다.

## 정리

바이브 코딩은 "AI가 코드를 대신 써준다"가 전부가 아니다. **AI와 내가 오래 같이 일할 수 있는 구조를 먼저 세우는 일**에 가깝다. 데이터와 화면을 분리해두면, 한 달 뒤의 나도, 그때의 AI도 같은 리듬으로 이어서 만들 수 있다.

다음 프로젝트에서도 나는 또 같은 질문부터 할 것이다. "이거, 데이터만 고치면 되게 만들 수 있나?"
