이 사이트(junominu.com)는 원래 도구 모음 허브였다. 어느 날 "내가 만든 것들을 한곳에 모아 보여주는 작업실"로 방향을 틀면서, 사이트 자체도 바이브 코딩으로 다시 키우기 시작했다. 이 글은 그 과정의 기록이다.

## 시작은 "데이터만 고치면 되는" 구조

가장 먼저 정한 원칙은 하나였다. **새 프로젝트가 생기면 코드가 아니라 데이터 한 줄만 추가하면 카드가 자동으로 생기게 한다.**

```js
{
  name: "프로젝트 이름",
  emoji: "✨",
  description: "한 줄 설명.",
  url: "https://...",
  status: "live",       // live | building | idea
  tags: ["태그1", "태그2"],
}
```

`src/projects.js`의 배열에 객체 하나를 넣으면 끝. 상태값(`live`/`building`/`idea`)에 따라 색이 다른 pill이 붙고, `url`이 없으면 클릭 안 되는 "구상 카드"가 된다. 화면(`Home.jsx`)은 그저 이 배열을 그릴 뿐이다. 만드는 사람 입장에서 마찰이 거의 없다는 게 핵심이었다.

## "만들었어요"에서 "이렇게 만들었어요"로

카드만 있으면 "이거 만들었어요(링크)"에서 끝난다. 그게 아쉬웠다. 그래서 두 번째 단계로 **프로젝트 상세 페이지(메이킹 스토리)**를 붙였다.

- `react-router`를 도입해 홈(`/`)과 상세(`/p/:slug`)를 나눴다.
- 상세 내용도 결국 데이터다. 각 프로젝트에 `detail` 객체(요약·기능·스택·타임라인·막혔던 점·핵심 프롬프트·회고)를 넣으면, 상세 페이지가 **있는 필드만 골라 자동으로 렌더**한다.
- 채우고 싶은 것만 채우면 되니, 글쓰기 부담이 줄었다.

여기서 한 가지 욕심을 더 냈다. 타임라인을 손으로 지어내지 않고 **실제 git 커밋 이력에서 뽑는** 것. 예를 들어 한 프로젝트는 첫 커밋이 2026-02-18, 약 3개월에 181커밋이었다. 그 마일스톤을 그대로 타임라인에 옮기니, 숫자가 거짓말을 하지 않는다. (이 과정에서 한 프로젝트의 스택을 "Next.js"로 잘못 적어둔 걸 발견하고 실제 `package.json` 기준으로 고치기도 했다. 기억보다 기록이 정확하다.)

## 그리고 이 블로그

세 번째 단계가 지금 읽고 있는 이 블로그다. 무겁게 가지 않았다.

- `src/posts/`에 마크다운 파일을 하나 넣으면 글이 된다.
- Vite의 `import.meta.glob`로 폴더의 `.md`를 전부 읽어 frontmatter(제목·날짜·요약·태그)를 파싱하고, `marked`로 본문을 HTML로 바꾼다.
- 목록(`/blog`)과 글(`/blog/:slug`) 두 라우트면 충분하다.

검색 유입을 위해 메타 태그와 OG, 사이트맵 같은 기본 SEO도 같이 깔았다. 글은 쌓일수록 복리라는 말을 믿어보기로 했다.

## 무엇을 배웠나

- **빼는 결정이 더 어렵다.** 기능을 더하는 건 쉽다. "이건 안 넣는다"를 정하는 게 사이트를 가볍게 유지하는 힘이었다.
- **데이터와 화면을 분리하면 운영이 쉽다.** 카드든 상세든 블로그든, 결국 "데이터 한 곳을 고치면 화면이 따라온다"로 통일했다. 다음에 뭘 추가할 때도 같은 리듬으로 간다.
- **기록이 콘텐츠가 된다.** git 이력, 막혔던 점, 실제로 썼던 프롬프트 — 만들면서 남긴 부스러기들이 그대로 이 사이트의 재료가 됐다.

비개발자가 AI 코딩으로 혼자 기획·개발·운영을 한다는 건, 결국 이런 작은 구조를 하나씩 세워가는 일이다. 이 사이트가 그 과정을 계속 기록하는 곳이 되면 좋겠다.
